### 真因特定 — `__file__`未定義
以前のセッションでダウンロード済みだったagentのstderrログ
（`episode-85020939-agent-{0,1}-logs.json`、Kaggle API
`competition_episode_agent_logs`で取得可能）に、実際のスタックトレースが残っていた:
```
File "/kaggle_simulations/agent/main.py", line 215, in <module>
    _HERE = os.path.dirname(os.path.abspath(__file__))
NameError: name '__file__' is not defined
```
**Kaggleの実行系（`kaggle_environments.agent.get_last_callable`）は`main.py`を
ソーステキストとして`exec(code, env)`で直接execする**。通常の`import`や
`importlib.spec_from_file_location`と異なり、この方式では`__file__`が一切定義
されない。`npmcts_policy.py`は重みファイルの場所特定に`__file__`を使っており、
**`agent()`が定義される前のモジュールレベルで即座にクラッシュ**していた。
numpy除去・act速度修正はどちらも正しい修正だったが、4連続エラーの真因ではなかった。

**なぜローカル検証で見逃したか**: `sandbox_replica.py`は
`importlib.util.spec_from_file_location`+`exec_module`でmain.pyを読み込んでおり、
これは通常のモジュールインポートと同じ扱いで`__file__`が自動的に設定される。
実際のKaggleローダー（`__file__`なしのbare exec）を再現できていなかった。

### 修正と最終結果
- `npmcts_policy.py`から`__file__`依存を完全除去、重みパスをdeck.csvと同じ相対パス
  方式（`"weights_pure.pkl"`→`/kaggle_simulations/agent/...`フォールバック）に統一。
- `sandbox_replica.py`を`exec(compile(src,...), module.__dict__)`方式に書き換え、
  実際のローダーを正確に再現するよう恒久修正（今後この種のバグを検出可能に）。
- **v015-fix4として再提出 → SubmissionStatus.COMPLETE。RLエージェントがラダーに
  無事投入された**（μ600から収束開始）。

本日はKaggle提出枠（5/日）を使い切った（v015-fix, v015-fix2, diag-probe, v015-fix3,
v015-fix4）。診断プローブ（v014クローン）が副産物としてラダーに乗っており、
eligibleペアは{v015-fix4, diag-probe}。
