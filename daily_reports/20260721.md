## 原因
`eval_both_bands.py` は **`sys.argv` に `--crn` がある時だけ** CRNハーネスを選択する設計:
```python

else:       from harness import run_gauntlet, load_engine
load_engine()
```
本日書いた評価スクリプトは**どれも `--crn` を渡していなかった**ため、
1. `eval_both_bands` が**素のエンジン**（`data/sim_sample/cg`）を先にキャッシュ
2. その後 `from harness_crn import run_gauntlet` しても、`cg` は既にキャッシュ済み
3. ⇒ `CG_CRN_SEED` を読む**パッチ版 `libcg.so`（exp052_crn/cg）は一度もロードされない**
4. ⇒ 「ペアCRN」と称した測定は**すべて無シード・無ペアリング**だった

修正（`sys.argv.append("--crn")` を import 前に実行）後、同一エージェントの再実行は
**0.8000 / 0.8000 で完全一致**を確認。

1. **「CRNを使っている」と称する測定は、実際に効いていることを毎回検証せよ**。
   最小コストの検証＝**同一エージェント・同一シードで2回走らせて完全一致するか**。

2. モジュールキャッシュ由来の失敗は**沈黙する**。`import` の順序が測定系を静かに壊した。
   スクリプト冒頭で**実際にロードされたエンジンのパスを assert** する（本日 `eval_bundle.py` /
   `eval_nodun.py` に実装済み）。
3. **提出前のビルド成果物検証がバグを捕まえた**。「モンキーパッチと実ビルドの一致確認」は
   出荷規律として維持する価値が実証された（これがなければノイズを提出していた）。
