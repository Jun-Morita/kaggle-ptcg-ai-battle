# exp041 — AlphaGo型パイプライン: 強パイロット対局データで事前学習 → MCTS → 自己対戦は最後

## 背景: RL失敗9件の根本原因整理（2026-07-07、ユーザーと合意）

silver目標・RL早期実装が勝負どころ、という前提で過去の失敗を機構別に棚卸しした結果
（詳細な議論はセッションログ、要約は memory `rl-status.md`）:

1. **学習データの分布（最重要）**: exp004/exp040の自己対戦は「弱いtraineeが生成したデータ」から
   学ぶ構造。crustleシャットアウトの機構が決定的な証拠——評価勝率87回連続0.000 ⇒ 学習中も
   crustleにほぼ勝てていない ⇒ crustle戦サンプルはほぼ全て負けラベル ⇒「crustleには負ける」
   しか学べない。Stage4の損失2倍重みが無効だったのは「負け」信号を2倍にしただけだから。
   一方**v014はcrustleに0.905で勝つ**＝勝ちトラジェクトリは自前で無限に生成できるのに未使用だった。
2. **方策priorのcold-start**: MCTSのpriorがほぼ一様のままでは search_count=16 の探索は実質
   ランダムウォーク。教師プールは相手側だけ有能にし、trainee側のpriorは未初期化のままだった。
3. **統合の近視眼**: exp033でAUC 0.806の価値ですらhorizon-1 greedy上書きはdoom-vetoに負け、
   壁(crustle)で悪化。exp010/040の「探索を深くするほど悪化」も同族（価値の質が探索量に見合わない）。
4. **データ規模の閾値（条件付きで解決済み）**: exp014→exp032で「10万試合・248万行あれば中盤AUC
   0.784」と確定。ただしexp032の成功条件は「**有能なパイロット同士の完走ゲーム＋最終勝敗ラベル**」
   であり、exp040の「弱いtraineeの手番のみ＋TD(λ)」とはデータの質が違う。
5. **表現の限界（未確定の範囲あり）**: exp022系のaction-match 0.49天井は「他人のエキスパート・
   別デッキ・2752決定」という悪条件下の結果。「自前パイロット・自前デッキ・無制限データ」での
   模倣可能性は**未検証**——本実験のPhase 2がこれを直接テストする。

解決済み資産: 決定化（exp019/exp040 Stage0, 85-96%厳密）、ネイティブエンジン（exp032, 0.12s/game）、
numpy蒸留（exp033）、公式transformer＋MCTS実装（exp040, opp_deck条件付き）。

## 仮説

AlphaZero型（ゼロから自己対戦）をやめ、**AlphaGo型（教師あり初期化→探索→最後に自己対戦）**に
切り替えれば、原因1・2・4を一挙に解決できる。核心は「学習データの生成を弱いtraineeから切り離し、
既に持っている強いルールベース（v014 turnbeam / revenge）に生成させる」こと。

## フェーズ設計（各フェーズに安い足切りゲート）

### Phase 1: データ生成（本ファイルの実装対象）
- `datagen_bc.py`: charmq側=v014(turnbeam)またはrevengeが操縦、相手=確立済み5マッチアップ場
  （crustle/ex_lucario/dragapult/archaludon/mirror）。exp032のdatagenパターンを踏襲しつつ、
  価値スカラーではなく**公式transformerの学習に直接使える形式**で記録:
  (sv_enc(opp_deck条件付き), sv_dec(全候補), 候補数, 選択候補index, 最終勝敗±1, matchup)。
- 候補列挙は exp040 `create_node` と同一ロジック（`enumerate_candidates`として抽出・共有）。
  パイロットの選択が列挙候補に無い場合（64候補cap超過 / minCount<maxCountの部分選択）は
  スキップしてカウント＝**公式サンプルの行動空間の限界を定量化**する副産物。
- 引き分けはexp032同様スキップ（±1ラベルを綺麗に保つ）。seatは1ゲームごとに交代。
- ゲート: スモーク(数十試合)でskip率・サンプル数/試合・速度・ファイルサイズを実測してから
  バルク規模（目標~100万サンプル程度）を決める。

### Phase 2: 教師あり事前学習 ＋ 決定的な安いゲート
- 公式transformer（exp040のMyModel, opp_deck条件付き）のpolicy headをBC（選択候補を正、
  他候補を負のadvantage目標）、value headを最終勝敗で学習。
- **ゲート（このライン全体の生死を決める）**: 探索なしの生ネット（policy argmax）が元パイロットに
  近い勝率を出せるか。目安: 各マッチアップでパイロット勝率の7割以上、特にcrustle>0.5。
  - 出せない → 「このアーキテクチャは自前ヒューリスティックすら表現できない」という決定的
    ネガティブ（原因5が確定）。ライン全体を安く畳む。
  - 出せる → 原因5の懸念を棄却、Phase 3へ。
- 副ゲート: 中盤価値AUC≥0.75（exp032の水準）。

### Phase 3: MCTS載せ
- exp040の修正済み決定化＋MCTSを事前学習済みネットの上に載せる。
- ゲート: 生ネットを上回る＆v014基準(総合2.67)にどこかのマッチで迫る。
  「探索を足すと悪化」が再発したら価値の質が不足（原因3）＝Phase 2に戻るか撤退。

### Phase 4: 自己対戦fine-tune（有能な初期値からのみ）
- exp040のselfplay/teacher-poolループを事前学習済み重みから再開。ここで初めてAlphaZero的
  ループが意味を持つ（勝ちトラジェクトリを自力生成できる初期値があるため）。

### 出荷経路（Phase 3以降で見えてきたら）
- numpy蒸留（exp033パターン）＋v013/v014の「検証付き上書き」哲学で統合（単独で打たせない）。
- 提出時は相手デッキ未知のため、opp_deck条件はexp038/039のアーキタイプ検出で代替する必要
  （oracle→推定のギャップはPhase 3で影響を測る）。
- 出荷判定は従来通り: v014合計(2.67)を明確な差で上回り、全マッチ無壊滅・0クラッシュ。

### トップランカーリプレイの位置づけ
補助。取得可能規模は数百試合（exp014で319試合）で原因4の閾値に届かない。まず無制限に生成できる
自前パイロットデータで土台を作り、上位者の勝ち筋は後段の味付け（fine-tune素材）として検討。

## Phase 2 実装ログ

### 実装(2026-07-07、datagen実行中に並行実施)
- `pretrain.py`: 公式transformer(d_model=128, heads=2, ff=256, enc/dec各1層=exp040のmain()と
  同一構成)の教師あり事前学習。
  - **ゲーム単位のtrain/val分割**(hash(worker,game_idx)%20==0がval=5%)——exp014のリーク教訓。
  - **ストリーミング学習**: 全ロードはRAM 10GB超になるため、毎エポックpickleチャンクを
    ディスクから流し読み(ファイル順+チャンク内shuffleの近似シャッフル)。
  - 損失はexp040 train()と同形式のHuber(value delta=0.2 / policy delta=0.1)、
    **matchup重みは中立**(Stage4の教訓: 重みは正例の欠如を直せない。今は正例があるので不要)。
  - BC policyターゲット: 選択候補+1.0/他-1.0(tanhヘッド、下流のMCTS prior∝exp(10p))。
  - val指標: policy top-1 accuracy(全体/matchup別/複数候補のみ) + 価値AUC(ゲーム内位相4分位、
    exp032のゲートは特に中盤=q2/q3)。
- `eval_raw.py`: **決定的ゲート**。生ネット(policy argmax、探索なし)を5マッチアップ場で対局させ、
  パイロット基準(v014: crustle 0.905/ex_lucario 0.77/dragapult 0.17/archaludon 0.195/
  mirror 0.585、合計2.67)と比較。生ネット≈パイロットなら表現力の懸念(原因5)を棄却してPhase 3へ、
  生ネット<<パイロットなら「無制限の同デッキデータでも公式アーキは自前ヒューリスティックを
  表現できない」という決定的ネガティブでライン全体を安く畳む。

### v015-fix3も再度ERROR → 真の根本原因を発見（`__file__`未定義、2026-07-10）
fix2の速度修正後もERRORが継続。診断プローブ（v014 main.pyそのまま＋未使用48MB重み同梱）を
提出したところ**COMPLETE**（ラダーに乗った、score 812.5）＝tarサイズ・大きな同梱ファイル
自体は無罪と判明。フォールバック完全対応版のfix3（重みをcg/内配置、import不死化、遅act
恒久停止スイッチ）を提出したが**それでも再度ERROR**。

replay APIから該当エピソードの完全なJSONを取得（`kaggle competitions episode-agent-logs`/
`episode-replay`のPython API、`competition_episode_replay`/`competition_episode_agent_logs`）。
**agentログに実際のスタックトレースが残っていた**（以前のセッションでダウンロード済みの
`episode-85020939-agent-{0,1}-logs.json`）:
```
File "/kaggle_simulations/agent/main.py", line 215, in <module>
    _HERE = os.path.dirname(os.path.abspath(__file__))
NameError: name '__file__' is not defined
```
**根本原因**: Kaggleの実行環境(`kaggle_environments.agent.get_last_callable`)は`main.py`を
`exec(code_object, env)`で**ソーステキストとして直接exec**する。通常の`import`/
`importlib.spec_from_file_location`と異なり、この実行方式では**`__file__`が一切定義されない**。
`npmcts_policy.py`は重みファイルの場所特定に`os.path.dirname(os.path.abspath(__file__))`を
使用しており、これがimport時（`agent()`が定義される前）に即座にcrashしていた。
**v015・fix・fix2・fix3の4連続エラーは全て同一原因**（numpy除去やact速度修正は正しい修正
だったが、真因ではなかった）。

**なぜローカル検証で見逃したか**: `sandbox_replica.py`は`importlib.util.spec_from_file_location`
+`exec_module`でmain.pyを読み込んでいた——これは**通常のモジュールインポートと同じ扱い**で
`__file__`が自動的に設定されるため、Kaggle実際の`exec(code, env)`（`__file__`なし）を
再現できていなかった。

**修正**: (1) `npmcts_policy.py`から`__file__`依存を完全除去、重みパスをdeck.csvと同じ
相対パス方式（`"weights_pure.pkl"`→`/kaggle_simulations/agent/...`フォールバック）に変更。
(2) `sandbox_replica.py`を実際のローダーと同じ`exec(compile(src,...), module.__dict__)`
方式に書き換え、`__file__`なし環境を正確に再現するよう恒久修正。

**再検証**: 修正後の`exec`再現ドライバで222手のフルミラー自己対戦を22.2秒で完走
（旧ドライバでは見逃していたクラッシュを新ドライバは正しく検出できることも別途確認）。
スモークn=8: 0エラー、3.61s/game。v015-fix4として再提出（ユーザー承認済み）。

**恒久教訓**: `__file__`はローカルテスト方法によって挙動が変わる罠。提出物の自己参照パスは
**deck.csvが最初から使っている相対パス＋`/kaggle_simulations/agent/`フォールバック方式**に
統一し、`__file__`は一切使わない。ローカル検証ドライバは実際のロード機構（bare exec）を
厳密に再現しなければ、この種のクラッシュを原理的に検出できない。
