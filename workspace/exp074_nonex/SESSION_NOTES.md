# exp074 — non_ex_attackers の解剖: プール最大の乖離を修理できるか

日付: 2026-07-21 〜 07-22
仮説番号: **H5**（「多くの仮説を出して検証する」方針の起点）

## 動機

exp073 が 452 実戦を使ってプール予測と実戦を突き合わせた結果、7アーキタイプ中
6つは「マッチメイキング圧縮」（似たレートの相手と当たるので勝率が 0.5 に寄る）で
説明できたが、**1つだけ符号が反転**した:

    non_ex_attackers   プール 0.847  →  実戦 0.259

圧縮は 0.847 を 0.5 の下には運べない。つまりこれは本物の乖離であり、
**プールが我々の最大の弱点を強みとして報告している**。

計測器が壊れたまま最適化しても無駄なので、H1〜H4（ミル高速化・副次プライズ等）
より先に H5＝計測器の修理を実行した。

## Stage 1 — 実デッキ・実パイロットの特定

実リプレイ 27試合（koff系4提出）を解剖。**7W-20L = 0.259** を再現。

単一クロック機構（exp067）は完全に一致した:

| | 相手デッキ残 | 我々のデッキ残 | **我々のプライズ残** | 相手のプライズ残 | 我々のターン数 |
|---|---|---|---|---|---|
| 勝ち | 0.29 | — | 6.00 | — | 17.43 |
| 負け | **12.60** | — | **5.90** | 1.70 | **13.95** |

我々はプライズをほぼ1枚も取らず、ミルだけで勝とうとして 12.6枚残して力尽きる。

**構造的原因は自前の知識ベースに既にあった**:
`references/knowledge/ptcg_strategy.md` — **「Safeguard / Neutralization Zone は
ex の攻撃のみ無効化」**。非exアタッカー相手には我々の防御技術は何も効かない。
ポケカの三すくみ（非ex ＞ {ex, 壁}）の頂点に対し、我々は餌食側にいる。

**外れた仮説（正直な記録）**: 「Abra/Kadabra が v030 で直した ex 祖先誤検出を
引き起こしている」— 誤り。これらは祖先集合に含まれない。敗因は誤検出ではない。

## Stage 1b — どのパイロットが実戦の勝率を再現するか（n=200, CRN）

実デッキ（最頻リスト）を抽出し、手持ちの全パイロットで測った:

| 相手パイロット | 我々の勝率 |
|---|---|
| 合成デッキ + RVP（**現行プール**） | 0.847 |
| **実**デッキ + RVP | 0.820 |
| **実**デッキ + AC generic | 0.820 |
| **実**デッキ + pub1034（探索付き） | **0.610** |
| 実戦 | 0.259 |

**デッキ差はわずか 0.027、パイロット差が 0.210。** それでも 0.26 の残差が残る。
→ 「実デッキに差し替えれば直る」という当初の H5 の想定は**否定された**。

## Stage 2 — バケットは1つのデッキではなかった（決定的発見）

リスト単位で戦績を出したところ、`non_ex_attackers` ラベルは**別々のデッキの寄せ集め**だった:

| ファミリー | 戦績 | 勝率 | 敗北数 | プールでの扱い |
|---|---|---|---|---|
| Alakazam 系 | 7-20 | 0.350 | 13 | `alakazam_dun` |
| **Dipplin 系** | **0-6** | **0.000** | **6** | **未モデル化** |
| その他 (Hop's Phantump) | 0-1 | 0.000 | 1 | 未モデル化 |

**訂正**: exp073 の「実戦 0.259」はバケット全体の値であり、プールが模した
Alakazam 系だけなら **0.350**。乖離は 0.588 ではなく 0.497（依然として巨大）。

さらに Alakazam 系の中で、**トップ7枚が同一なのに 0-6 のリストと 4-4 のリストが
共存**していた。差分は Dunsparce の版だけ（cardId 65 = HP60 / 305 = HP70）。
n=6 と n=4 なので断定はできないが、デッキ差より**操縦とばらつき**が支配的である
ことを示唆する。

Dipplin 系（実物、0-6 の相手）:

```
x5 Basic {G} Energy / x4 Grookey / x4 Thwackey / x4 Applin / x4 Dipplin
x4 Lillie's Determination / x4 Buddy-Buddy Poffin / x4 Poké Pad
x4 Bug Catching Set / x4 Festival Grounds / x2 Kieran / x2 Boss's Orders

x1 Rellor / x1 Rabsca / x1 Shaymin / x1 Black Belt's Training
x1 Lana's Aid / x1 Switch / x1 Secret Box
```

**未モデル化の 0.000 マッチアップは、誤予測のマッチアップより大きな計測器の穴。**
これが H5 から派生した新仮説 **H6** になる。

## Stage 3 — Dipplin の局所測定（n=200, CRN, エラー0）

| 相手パイロット | 我々の勝率 |
|---|---|
| RVP | 0.740 (148-52) |
| AC generic | 0.740 (148-52) |
| pub1034 | 0.615 (123-77) |
| 実戦 | 0.000 (0-6) |

**RVP と AC が完全に同一の戦績（148-52-0）**— CRN バグを暴いたのと同じシグネチャ。
調査したところ、両者は同じ基底方策 `policies/lucario_v2.py` を読み込んでおり、
RVP が追加する `PATCH_SRC` は**デッキ依存ディスパッチ**（Trevenant / Hop's 非ex系）
なので Alakazam や Dipplin には効かない。つまりこれらのデッキでは RVP ≡ AC。

## Stage 4 — H7: プール全体が「1つの相手の脳」だった

`opponents()` を**デッキ別ではなくパイロット別**に監査した:

| パイロット | スロット | V3 重み合計 |
|---|---|---|
| **lucario_v2** | lucario_ex, pure_wall, alakazam, alakazam_dun, marnie, cinderace | **86.8%** |
| その他 | crustle_LO, archaludon, dragapult | 13.2% |

**バンド重みの 86.8% が単一のパイロットで駆動されていた。** 今年の全ゲート、
全 CMA-ES チューニング（exp058/exp060）、全採用判定が、**我々が何ヶ月も暗黙に
フィットさせてきた1つの相手の脳**に対して採点されていた。教科書的な単一相手への
過学習であり、観測される楽観バイアスをそのまま予測する:

| | 値 |
|---|---|
| プール（バンド加重） | 0.7661 |
| 実戦（147試合） | 0.5578 |
| **差** | **0.2083** |
| Stage 1 のマッチ単位パイロット差（0.820→0.610） | **0.2100** |

**2つの差が 0.002 の精度で一致する。** これが H7 を重く見る理由で、
プール→実戦の誤差はアーキタイプ重み（V2/V3 で2回較正済み）ではなく
**パイロットのモノカルチャー**が原因だと言っている。

`eval_onepilot.py` — `eval_both_bands.py` は一切変更せず、手持ちで唯一
本質的に異なり強い脳（pub1034 の探索付きパイロット）で全デッキを操縦して再採点。
事前登録した読み: 0.558 付近に着地すれば H7 確定、0.766 のままなら H7 棄却。

スモーク n=10（エラー0）:

| スロット | 重み | lucario_v2 | pub1034 | Δ |
|---|---|---|---|---|
| alakazam_dun | 0.2891 | 0.847 | 0.600 | −0.247 |
| marnie | 0.1600 | 0.912 | 0.600 | −0.312 |
| alakazam | 0.1239 | 0.893 | 0.500 | −0.393 |
| **pure_wall** | 0.0933 | 0.205 | 0.600 | **+0.395** |
| lucario_ex | 0.0670 | 0.785 | 0.700 | −0.085 |

バンド 0.7661 → 0.6020、実戦との残差 **+0.2083 → +0.0442**。

**注意（一律の再操縦は正しくない）**: `pure_wall` だけ符号が逆で、勝率が上がる。
pub1034 は壁デッキのパイロットではないため、この枠は現行の `AC.make_crustle_agent`
の方が忠実な代理である可能性が高い。**スロットごとに最も忠実なパイロットを選ぶ**
必要がある（＝ユーザー提案のハイブリッド方式そのもの）。n=200 の結果を待って確定。

## 教訓

1. **アーキタイプ・ラベルはプールのスロットとして粗すぎる。** 1ラベル＝1デッキ＝
   1パイロットという仮定が、実際には3つの異なるデッキを1つのスロットに潰していた。
2. **パイロット忠実度がプール誤差の主因**（0.210 / 0.237）。これは exp072 で
   ユーザーが指摘した「同一デッキが増えるとパイロットの優劣の重要性が増す」と
   同じ結論に、別の経路から到達している。
3. 残差 0.26 は Stage 4 で説明がついた —— **プールの 86.8% が単一パイロット**。
4. **同一の戦績が2度出たら疑う。** CRN バグも本件も、最初の手がかりは
   「違うはずのものが完全に同じ数字を返した」ことだった。異なる設定が
   ビット単位で一致したら、それは偶然ではなく配線の証拠。
5. プールの信頼性を上げる作業は、これまで最も歩留まりが高いレーン
   （V2/V3 較正、CRN バグ、述語の真値採点）を一貫して継続している。

## Stage 5 — H2（サイドを取る二本目のクロック）: 棄却

`oracle_prize.py` でハーネス側からオラクル介入（技選択で最大打点を強制、
リーサルがあれば MAIN で攻撃を強制）。**上限 +0.0000**、オラクルは一度も発火せず。

原因は「選ばない」ではなく「**選べない**」だった:

- 4試合で提示された攻撃は**6回、すべて Land Collapse（0ダメージ）**。
  Superb Scissors(120/3エネ)、Giant Tusk(160/4)、Land Crush(90/3) は一度も出現しない。
- 我々のバトル場のエネルギー数:

| 相手 | 0個 | 1個 | 2個 | 3個以上 |
|---|---|---|---|---|
| alakazam_dun | 26T | 10 | 1 | **0** |
| pure_wall | 42T | 19 | 12 | **0** |

デッキのエネルギーは60枚中8枚（Rock Fighting x4 + Mist x4）。3個は構造的に不可能。
**H2 は方策では不可能。デッキ変更なしには成立しない。**

## Stage 6 — H1（ミル高速化）: 数値化

最初の `probe_mill.py` は相手の `deckCount` 減少をミルとみなし、相手自身のドローと
混同して 1試合48〜60枚という不合理な値を出した。**破棄**。ログ（`LogType.ATTACK` と
相手 DECK→DISCARD の `MOVE_CARD`）から取り直したのが `probe_mill2.py`。

```
ミル発生 209件:  1枚 = 172 (82.3%)   4枚 = 37 (17.7%)
```

Land Collapse は通常1枚、**そのターンに古代サポートを使えば4枚**。
古代フラグは CardData に無いため、どのサポートが古代かを実測で特定した:

| カード | ブースト時 | 非ブースト時 |
|---|---|---|
| **Explorer's Guidance** | 29 | 0 |
| Colress's Tenacity | 0 | 11 |
| Boss's Orders | 0 | 11 |
| Lisia's Appeal | 0 | 5 |

**古代サポートは Explorer's Guidance のみ、4枚。**

### 無料の修正（優先順位）は無かった

サポートは1ターン1枚。`probe_eg.py` で攻撃ターンを分類（n=257）:

```
ブースト成功                        26  10.1%
EG が手札にあったが別のサポートを使用  1   0.4%
EG が手札にあったがサポート未使用     13   5.1%
EG が手札に無い                    217  84.4%
```

取りこぼしは 5.4%（+0.16枚/攻撃）のみ。**方策は既に EG を優先できている。**
ブーストは優先順位ではなく**引き**に律速。古代サポートを4→8枚にしても
per-attack は 1.53→1.9 程度で、必要な +0.9枚/ターンには届かない。

## Stage 8 — プールの Alakazam 枠は相手の自滅で決まっていた

決着理由を直接確認（各12試合）:

```
alakazam_dun 11/12 勝  — 全勝ちが opDeck=0（相手の山札切れ）
                          1件は turn=4 で opActive=False/opBench=0（場のポケモン切れ）
alakazam     12/12 勝  — 同上
```

我々のミル貢献は 1試合 2.4枚。**残り約50枚は相手が自分でドローして掘り切っている。**
lucario_v2 が Alakazam デッキを操縦してドローサポートで自分の山を燃やしている。

**これが H7 の試合レベルの裏付け。** プール重みの 41% が相手パイロットの自滅で
決まっており、実際の Alakazam 使いは自滅しないので 0.847 → 実戦 0.350 になる。

**実戦の Alakazam 系に対して我々には勝ち筋が無い**（ミル1.4〜2.4枚、サイド0枚）。
